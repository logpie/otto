Fix the cache stampede bug in cachetools function decorators (ttl_cache, lru_cache, etc.). The bug: when many threads simultaneously call a decorated function with the same key before the first call returns, ALL threads execute the underlying function instead of only one. With 100 threads calling the same key on a function that takes 0.05s, the expected result is CacheInfo(hits=99, misses=1) but the actual is CacheInfo(hits=0, misses=100). The fix must satisfy two requirements: (1) for the same key, only one thread computes the value while others wait and then receive the cached result, and (2) calls with DIFFERENT keys must still run concurrently — do not sacrifice parallelism for correctness. Add a threading test that verifies both properties. Run tests with: pytest
