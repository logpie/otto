Fix the cache stampede bug in cachetools function decorators (ttl_cache, lru_cache, etc.). The bug: when many threads simultaneously call a decorated function before the first call returns, ALL threads get cache misses instead of only one. With 1000 threads calling a function that sleeps 0.1s, the expected result is CacheInfo(hits=999, misses=1) but the actual is CacheInfo(hits=0, misses=1000). The lock is released before the value is stored in the cache, allowing all threads to execute the underlying function. Fix the lock/cache interaction so only one thread executes the function while others wait for the cached result. Add a threading test that verifies: spawn 100 threads calling the same cached function simultaneously, assert misses=1 and hits=99. Run tests with: pytest
