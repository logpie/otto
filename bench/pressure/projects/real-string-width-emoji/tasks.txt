The string-width library incorrectly calculates the width of minimally-qualified and unqualified emoji sequences. For example, keycap sequences like "1\u20E3" (digit one + combining enclosing keycap, without VS16) render as double-width emoji in terminals but string-width reports them as width 1. Similarly, ZWJ emoji sequences that lack VS16 qualifiers (like "\u{1F468}\u200D\u{1F469}\u200D\u{1F467}" without the \uFE0F characters) are not detected by the \p{RGI_Emoji} regex because they're "minimally qualified", so they fall through to East Asian Width logic which returns incorrect widths. Fix index.js to detect these non-RGI but still double-width emoji sequences: keycap sequences (digit/hash/asterisk + U+20E3) and ZWJ sequences with 2+ Extended_Pictographic code points.
