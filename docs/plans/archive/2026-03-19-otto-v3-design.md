# Otto v3 — Reliability Harness Architecture

**Date**: 2026-03-19
**Status**: Implemented

## Context

Benchmarked otto v2 (adversarial TDD pipeline) against baselines:

| Scenario | bare-cc (single claude -p) | ralph (loop) | otto v2 (rubric→testgen→coding) |
|---|---|---|---|
| Easy (detailed spec) | 100%, $0.85, 3min | 100%, $0.73, 2min | 100%, $0.43*, 7min |
| Hard (detailed spec) | 100%, $0.66, 2min | 100%, $0.68, 2min | 50%, $1.35, 7min |
| Vague specs | 63% validator rate | 54% | 46% (worst) |

*Otto cost underreported — missing rubric+testgen costs.

**Finding**: Otto's multi-agent pipeline (rubric→testgen→coding) is never better and often worse than a single unconstrained `claude -p` call. The sequential handoffs lose information. Telling the coding agent "don't write tests" removes its strongest self-correction tool.

## Mission

Fully autonomous coding agent that users trust to run unattended 24/7. Users specify intent and constraints. Results must be NO WORSE than standard `claude -p`.

## Architecture

Otto v3 is multi-agent but NOT a constraining pipeline. Agents work independently, coordinated by a reliability harness.

```
┌─────────────────────────────────────────────────────────┐
│                    otto v3                              │
│  "Reliability harness that makes Claude safe to run     │
│   unattended while you sleep"                          │
└──────────────┬──────────────────────────────────────────┘
               │
    ┌──────────▼──────────┐
    │   Pilot Agent       │   (LLM — strategic decisions)
    │                     │
    │ • Task ordering     │
    │ • Parallel/serial   │
    │ • Smart retry       │
    │ • Recovery/escalate │
    │ • Doom-loop detect  │
    └──────────┬──────────┘
               │
    ┌──────────▼────────────────────────────────┐
    │     Per-Task Loop                         │
    │                                           │
    │  1. Create branch (worktree if parallel)  │
    │  2. Coding Agent (UNCONSTRAINED)          │
    │     - reads codebase                      │
    │     - implements feature                  │
    │     - writes tests                        │
    │     - runs tests, fixes failures          │
    │  3. External verify (existing tests in    │
    │     clean worktree + custom verify cmd)   │
    │  4. Pass → squash merge                   │
    │     Fail → pilot decides retry strategy   │
    └──────────┬────────────────────────────────┘
               │
    ┌──────────▼──────────┐
    │  Post-Run Gates     │
    │                     │
    │ • Full regression   │
    │ • Review Agent      │
    │   (cross-task diff) │
    │ • Integration gate  │
    └─────────────────────┘
```

## Agents

### 1. Pilot Agent
LLM orchestrator that makes strategic decisions. Replaces deterministic pipeline with adaptive intelligence.

**Decisions:**
- Which tasks to run in parallel vs serial
- When to retry (and how — resume session vs fresh start)
- When to escalate (cost cap hit, doom loop detected, repeated failures)
- Whether to run architect phase (2+ tasks touching same files)
- Post-run: whether integration gate is needed

### 2. Coding Agent
Single unconstrained Claude agent (Agent SDK) that does everything:
- Reads codebase, understands structure
- Implements the feature
- Writes tests for the feature
- Runs all tests (own + existing)
- Fixes failures iteratively
- Decides when done

**Key change from v2**: NOT told "don't write tests." NOT constrained by external test files. Full autonomy over implementation approach.

**Prompt structure:**
```
{task spec}

You are working in {work_dir}.
{optional: architect conventions}
{optional: failure feedback from previous attempt}
{optional: relevant source context — pre-loaded to save exploration}
```

### 3. Architect Agent
Analyzes codebase and establishes shared conventions for multi-task runs. Only runs when 2+ tasks are pending.

**Outputs:**
- `conventions.md` — coding patterns, naming, error handling
- `file-plan.md` — predicted file conflicts, dependency injection

**Simplified from v2**: No test patterns, no conftest generation. The coding agent handles its own test conventions.

### 4. Review Agent
Post-merge cross-task review. Reads the combined diff of all tasks and checks for:
- Inconsistencies between tasks
- Conflicting changes to shared files
- Architectural issues

Advisory, not gating. Pilot decides whether to act on findings.

## Verification Strategy

Three layers, none generated by a separate agent:

1. **Agent's own tests** — the coding agent writes and runs tests during implementation. This is how bare-cc achieves 100% on easy and hard tasks.

2. **Existing test suite** — run in a clean disposable worktree after the agent finishes. Catches regressions. (For greenfield projects, this layer is empty initially — the agent's own tests are the primary signal.)

3. **Custom verify command** — user-provided validation. `otto add --verify "curl localhost:8080/health"`. For domain-specific checks that tests can't cover.

## Retry Strategy

```
Attempt 1: Fresh prompt with spec + source context
  → Coding agent runs (writes code + tests, runs tests internally)
  → External verification in clean worktree
  → FAIL: existing test_login broke

Attempt 2: Resume session with failure feedback
  → "Verification failed: test_login.py::test_session_expires FAILED\n{output}"
  → Coding agent fixes with full context from attempt 1
  → External verification
  → PASS: merge

Attempt 3+ (if needed): Fresh session
  → Don't resume — context may be polluted
  → Full spec + accumulated failure context
  → "Previous attempts failed because: {summary}"
```

**Pilot intelligence:**
- Same test failing twice → likely a real conflict, not a transient issue
- Cost exceeds threshold → stop, report partial progress
- Agent loops without making progress → doom-loop detection, abort

## Multi-Task Orchestration

```
Phase 0: Baseline check (existing tests pass before we start)
Phase 1: Architect analysis (optional, 2+ tasks)
Phase 2: Task execution
  - Topological order from dependency graph
  - Independent tasks run in parallel worktrees
  - Sequential merge after each parallel batch
Phase 3: Post-run
  - Full regression gate (all tests, one more time)
  - Review agent (combined diff)
  - Report results
```

## What Otto Adds Over Raw claude -p

| Capability | Raw claude -p | Otto v3 |
|---|---|---|
| Single task implementation | Full power | Same (unconstrained agent) |
| Branch isolation | None | Auto per-task branches |
| Multi-task orchestration | Manual | Dependency graph, parallel |
| Retry with external feedback | None | Verify in worktree → retry |
| Regression protection | None | Existing test suite gate |
| Cross-task consistency | N/A | Architect + review agent |
| Git safety | None | Worktrees, clean merges, stash |
| Cost/time tracking | None | Per-task and per-run metrics |
| Crash recovery | None | Session tracking, state persistence |
| Doom-loop detection | None | Cost caps, progress monitoring |

## What Gets Removed from v2

| v2 Component | Status | Reason |
|---|---|---|
| Rubric agent | Removed | Bench shows it loses information vs raw spec |
| Testgen agent | Removed | Bench shows adversarial tests hurt more than help |
| "Don't write tests" constraint | Removed | Bench shows self-testing is Claude's strongest tool |
| Test bug diagnosis agent | Removed | No adversarial tests = no test bugs to diagnose |
| Integration fix agent | Removed | Just retry coding agent with failure context |
| Holistic testgen | Removed | No testgen at all |
| Test validation | Removed | No generated tests to validate |
| Pilot MCP server | Simplified | Keep pilot decisions, simplify tool interface |

## Greenfield Project Considerations

For projects with no existing test suite:
- The coding agent's own tests are the primary quality signal
- The pilot should prioritize tasks that establish test infrastructure early
- `otto init` should suggest a verify command
- Over time, the growing test suite provides increasingly strong verification

## Risk Assessment

**Risk: Same agent marks its own homework (no adversarial tests)**
Mitigation: The bench data shows this works. bare-cc passes 100% of detailed-spec tasks. For greenfield, the review agent provides a second perspective. For production, users should provide verify commands.

**Risk: This makes otto a thin wrapper around claude -p**
Response: For single tasks, yes. The value is: branch isolation, verification gates, retry, crash recovery. For multi-task, the orchestration is substantial. Otto is infrastructure, not intelligence.

**Risk: Vague specs still produce poor results (all runners scored 0/8)**
Mitigation: The pilot can detect ambiguity and either proceed under explicit assumptions or ask clarifying questions. The coding agent's own judgment (63% bare-cc) is better than rubric-constrained judgment (46% otto v2).
