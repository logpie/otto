# Retry & Recovery UX Improvements

**Date:** 2026-03-15
**Status:** TODO

## Done (this session)

- `otto status` detects stale `running` tasks (lock check), shows actionable warning
- `otto run` auto-recovers stale `running` → `pending` at startup
- `otto run` auto-unblocks `blocked` tasks whose deps are now `pending`/`passed`

## TODO

### 1. `otto retry` without `--force` for stale tasks
`otto retry 3` should work without `--force` when the task is stuck in `running` and no otto process is active (same lock check as status). Currently requires `--force` because `running != failed`.

### 2. `otto retry` cascades to blocked dependents
When you `otto retry 1` (a failed task), tasks that were blocked by #1 should also be offered for reset:
```
✓ Reset task #1 to pending
  Tasks #2 and #4 were blocked by #1. Also reset? [Y/n]
```

### 3. `otto retry --run` combines retry + run
```
otto retry 3 --run    # reset to pending AND immediately run
```
Saves the two-step dance of `retry` then `run`.

### 4. `otto run <id>` to run a specific task
```
otto run 3            # run just task #3 (useful for debugging one task)
```
Currently `otto run <prompt>` creates an ad-hoc task. Could use `otto run --task 3` to avoid ambiguity with prompt.
