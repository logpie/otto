# Otto: Intent-to-Product v2 — Outer Loop Around v4.5

**Date**: 2026-03-29
**Status**: Design
**Supersedes**: `2026-03-26-otto-intent-to-product.md` (gates + gate pilot approach — abandoned after testing showed inner loop is already reliable enough)
**Builds on**: v4.5 pipeline (inner loop, implemented and proven)

---

## Motivation

Otto v4.5 is a reliable task executor. Users decompose their intent into tasks manually:

```bash
otto add "set up data model"
otto add "implement CRUD API"
otto add "build web UI"
otto run
```

This works, but it's low-level. Users have to think about architecture, task sizing, dependencies, and execution order. The coding agent can handle product-level prompts — `otto add "build a weather app like Apple's"` produced a working Next.js app in one shot ($2, 6 min). But this breaks at complexity. Multi-surface products (auth + payments + admin + API + frontend) can't fit in one agent session.

The gap: **no product-level interface that decomposes intent into tasks and verifies the result as a whole.**

---

## Design Principles

1. **Outer loop, thin.** i2p is a wrapper around v4.5, not a replacement. The inner loop (task-level execution) is unchanged.
2. **Single-task is a valid plan.** `otto build` should not force decomposition. If the intent fits one agent session, one task is the right plan.
3. **Scope control over planning quality.** The hard problem isn't "what tasks to create" — it's preventing scope creep and false confidence. Product spec defines what's IN and what's NOT.
4. **Verify against claims, not against vibes.** Product QA tests whether the delivered artifact satisfies the spec's explicit user journeys and acceptance criteria. Not "does it look good."
5. **Earn complexity with failure data.** No gates, no pilots, no multi-layer verification. Add these only when failure patterns prove they're needed.
6. **Bad decomposition is worse than no decomposition.** Horizontal slicing (data layer → logic layer → API → UI) means every task depends on the one before and integration bugs hide between layers. Vertical slicing (feature A end-to-end, feature B end-to-end) means each task is independently testable and deliverable. Example:

   Horizontal (fragile):
   ```
   Task 1: Build data models (models.py)
   Task 2: Build API layer (api.py) — can't test without task 1
   Task 3: Build UI (frontend/) — can't test without task 2
   ```

   Vertical (robust):
   ```
   Task 1: Bookmark CRUD — model + API + UI for creating/listing
   Task 2: Tag system — tag model + tag API + tag filtering UI
   Task 3: Search — search index + search endpoint + search UI
   ```

## The Outer Loop

```
otto build "bookmark manager with tags, search, and Chrome extension"
         │
         ▼
    ┌─────────┐
    │ CLASSIFY │  Is this a single-task job or does it need decomposition?
    └────┬────┘  If simple/cohesive → single task, skip planning
         │
         ▼
    ┌─────────┐
    │  PLAN   │  Product planner: spec + architecture (if needed) + tasks
    └────┬────┘  User reviews by default. --no-review to skip.
         │
         ▼
    ┌─────────┐
    │ EXECUTE │  v4.5 inner loop — unchanged
    └────┬────┘  Batching, parallel execution, per-task QA, merge
         │
         ▼
    ┌─────────┐
    │ VERIFY  │  Product QA: test user journeys from product spec
    └────┬────┘  Did the PRODUCT work, not just individual tasks?
         │
         ▼
    ┌──────────────┐
    │ FIX or REPLAN│  Failure triage:
    └──────────────┘  Implementation bug → fix task → re-execute → re-verify
                      Planning failure → replan → re-execute → re-verify
                      Scope issue → report to user
```

Five steps. One is the existing inner loop. Four are new.

---

## Step 1: Classify

Before planning, determine if decomposition is even needed.

**Decompose when** the intent contains multiple failure domains:
- Multiple user-facing surfaces (web + extension + API)
- Auth/payments/security (separate concern from features)
- Multiple external integrations
- Multiple user roles
- Background jobs / async processing

**Single task when** the intent is cohesive:
- One surface, one data model, one core workflow
- "Build a weather app" — one page, one API, one data source
- "Add search to the API" — one feature in existing codebase
- "Build a CLI tool that does X" — contained scope

The planner makes this call. Single-task plan skips decomposition entirely — just writes one task to tasks.yaml and proceeds.

```python
# Planner can output:
{"mode": "single_task", "task": "full prompt for the one task"}
# or:
{"mode": "decomposed", "product_spec": "...", "tasks": [...]}
```

---

## Step 2: Plan

One agent call. Produces:

### For single-task mode:
- One task in `tasks.yaml` with a comprehensive prompt
- No product-spec.md, no architecture.md (overkill for one task)

### For decomposed mode:
- **`product-spec.md`** at project root — features, scope, non-goals, user journeys (3-5). Committed to git.
- **`architecture.md`** at project root (only when meaningful) — tech stack, data model, API shape. Skip for simple projects where the coding agent can decide. Committed to git.
- **`tasks.yaml`** — tasks with dependencies, vertically sliced where possible

### Planner prompt (goal-oriented)

```
You are a product planner for an autonomous coding system.

Given a user's intent, decide:
1. Can this be built in ONE agent session? If yes, output a single comprehensive task.
2. If not, decompose into the FEWEST vertical tasks possible.

If decomposing, produce:
- product-spec.md: features, scope, NON-GOALS (critical — what NOT to build),
  and 3-5 user journeys written from the USER'S perspective.

  BAD journey: "search endpoint returns results" (that's a task spec)
  GOOD journey: "A user saves 20 bookmarks over a week, tags some with
  'work' and 'recipes', then searches for 'pasta' and finds the 3 recipe
  bookmarks they saved. They export all bookmarks, delete app data,
  re-import, and everything is still there."

  Journeys should span multiple features and reflect how a real person
  uses the product over time. They are the acceptance test for the
  whole product, not individual features.

- tasks.yaml: tasks with depends_on. Prefer vertical slices (feature end-to-end)
  over horizontal layers (data → logic → API → UI).
  Each task must be independently testable.

If the intent is ambiguous, list your assumptions. Do NOT silently fill gaps.

Architecture.md: only produce this when there are genuine tradeoffs
(multiple viable tech stacks, complex data model, external integrations).
For simple projects, let the coding agent decide.
```

### User review

Default: show the plan and wait for approval.

```
$ otto build "bookmark manager with tags, search, Chrome extension"

Product: Bookmark Manager
Scope: single-user, local SQLite, web UI + Chrome extension
Non-goals: multi-user, cloud sync, mobile app

Tasks (4):
  #1 Project setup + bookmark CRUD with tags (no deps)
  #2 Full-text search (depends: #1)
  #3 Web UI (depends: #1)
  #4 Chrome extension (depends: #1)

Estimated: ~$4-6, ~20 min

[enter] accept  [e] edit  [q] quit
```

`--no-review` skips this and runs immediately.

### What the planner should NOT do

- Invent features the user didn't ask for (scope creep)
- Choose architecture when the user specified one ("use Flask" → use Flask)
- Produce 10 tasks when 4 would do (over-decomposition)
- Separate "write tests" as a task (tests are part of each task)

## Step 3: Execute

**v4.5 inner loop, with one addition:** a persistent context file that accumulates knowledge across tasks.

### Product artifacts live in the project tree

```
project_root/
  ├── product-spec.md      # planner creates, user reviews, committed to git
  ├── architecture.md      # planner creates (when needed), committed to git
  ├── context.md           # auto-maintained after each task, gitignored
  ├── src/
  ├── tests/
  ├── otto.yaml
  ├── tasks.yaml
  └── otto_logs/           # otto runtime (gitignored)
```

These are **normal project files**. The coding agent discovers them by exploring the codebase — no special injection needed. It reads `architecture.md` the same way it reads `README.md`. Users can edit them, commit them, review them in PRs.

### Context file (accumulates across tasks)

`context.md` at project root. Auto-maintained, gitignored. Captures what the codebase alone doesn't tell you: WHY decisions were made, what conventions emerged, what gotchas were discovered.

**Why this is needed:** The codebase shows WHAT exists. It doesn't show:
- Why task 1 chose WAL mode for SQLite (task 3 might hit the same concurrency issue)
- That task 2 established `app.use('/api/<resource>', router)` as the route pattern (task 4 might register routes differently → merge conflict)
- That a previous task tried FTS5 and it wasn't available (task 5 might waste time trying it again)

Without this, each task rediscovers the project from scratch by reading code. At 5+ tasks, that's significant wasted work and a source of inconsistency.

**How it works:**

```
Task 1 completes → LLM reads the diff → updates context.md
Task 2 starts   → agent explores project, finds context.md, reads it
Task 2 completes → LLM reads the diff → updates context.md
Task 3 starts   → agent finds context.md (now richer)
...
```

One cheap LLM call (~$0.01) per task completion. The agent finds the file naturally — no prompt injection.

**What it contains:**

```markdown
# Product Context
<!-- Auto-updated after each task. DO NOT edit manually. -->

## Architecture Decisions
- SQLite with WAL mode — concurrent writes fail without it (task 1)
- Express routes: app.use('/api/<resource>', router) pattern (task 1)
- React frontend with Vite, pages in src/pages/ (task 3)

## Data Model (current state)
- bookmarks(id, url, title, description, created_at)
- tags(id, name) + bookmark_tags(bookmark_id, tag_id) (added by task 2)

## API Endpoints
- POST/GET/PATCH/DELETE /api/bookmarks (task 1)
- GET /api/tags, POST /api/bookmarks/:id/tags (task 2)
- GET /api/search?q=term (task 3)

## Conventions
- Tests in __tests__/ using vitest (not jest)
- Error responses: {error: string, status: number}
- Shared React components in src/components/

## Gotchas
- SQLite FTS5 not available in runtime — use LIKE queries (task 3 discovered)
- Node 18+ required — optional chaining used throughout (task 1)
- Chrome extension manifest v3 — no background pages, use service workers
```

**The update call:**

```python
async def update_product_context(project_dir: Path, task_key: str, diff: str, config: dict):
    """After a task completes, update context.md with discoveries."""
    context_path = project_dir / "context.md"
    existing = context_path.read_text() if context_path.exists() else ""

    prompt = f"""A coding task just completed. Update the product context file
with any new information.

Current product context:
{existing or "(empty — this is the first task)"}

Task that just completed: {task_key}
Diff:
{diff[:5000]}

Update the product context with:
- New architecture decisions (and WHY they were made)
- Data model changes (new tables, columns, relationships)
- New API endpoints
- Conventions the task established or followed
- Gotchas discovered (things that didn't work, workarounds needed)

Return the FULL updated context.md content.
Do NOT remove existing entries — only add or update."""

    updated, cost, duration = await _run_planner_prompt(prompt, config, project_dir, ...)
    context_path.write_text(updated)
```

**When to create:**
- `otto build` (decomposed mode, 3+ tasks): always
- `otto build` (single-task mode): skip
- `otto add` + `otto run` (manual multi-task): create if >2 tasks

**This is analogous to CC agent teams' shared context**, implemented as a project file instead of peer messaging. The codebase is the primary communication channel. The context file carries intent, conventions, and gotchas that code alone doesn't express.

## Step 4: Verify (Product QA)

After all tasks pass individually, verify the PRODUCT works as a whole.

### What it tests

Derived from `product-spec.md` user journeys — written from the user's perspective, spanning multiple features, reflecting how a real person uses the product over time.

```
Product spec says:
  Journey 1: Daily use — user saves 20 bookmarks over a week, tags some
    with "work" and "recipes", searches for "pasta", finds the 3 recipe
    bookmarks. Verifies search is fast and results are relevant.
  Journey 2: Cross-surface — user clicks Chrome extension on a recipe page,
    bookmark appears in web UI within seconds, tags are preserved.
  Journey 3: Data safety — user exports all bookmarks, deletes app data,
    re-imports the file, everything is restored exactly.
```

These journeys test what per-task QA cannot: features working TOGETHER as a product, data flowing across surfaces, state consistency over time.

Product QA agent receives: product-spec.md + architecture.md + full codebase. Tests each journey end-to-end using browser (chrome-devtools MCP) and/or API calls. Collects evidence (commands run, responses received, screenshots).

### What it produces

```json
{
  "product_passed": true,
  "journeys": [
    {"name": "Save & Search", "passed": true, "evidence": "..."},
    {"name": "Chrome Extension", "passed": false, "error": "extension popup loads but save button returns 404", "evidence": "..."},
    {"name": "Import", "passed": true, "evidence": "..."}
  ]
}
```

### When to skip

- Single-task mode with no product spec → skip (task-level QA is sufficient)
- `--no-qa` flag → skip
- All tasks were single-attempt passes and integration test passes → consider skipping (low risk)

### LLM as judge, not oracle

Product QA agent runs real commands (curl, browser automation, file checks) and collects evidence. It does NOT judge by reading code. Evidence-based verification only.

## Step 5: Fix or Replan

When product QA fails, classify the failure and respond accordingly.

### Implementation bug (most common)
The code has a bug that task-level QA missed because it only shows at integration.

```
Journey "Chrome Extension" failed:
  Extension save button calls POST /bookmarks
  API is at POST /api/bookmarks
  → 404
```

**Action:** Create a fix task, run through inner loop, re-verify.

```python
otto_add(f"Fix: Chrome extension calls POST /bookmarks but API is at POST /api/bookmarks. Update extension to use correct endpoint.")
# → runs through v4.5 → re-run product QA
```

### Planning failure (less common)
The plan was wrong — wrong architecture decision, missing task, bad decomposition.

```
Journey "Save & Search" failed:
  Search returns empty for all queries
  → SQLite FTS5 not available in runtime
  → Architecture chose FTS5 but the deployment target doesn't support it
```

**Action:** Revise the plan. Update architecture, re-decompose affected tasks, re-execute.

This is the "replan escape hatch" — not endless patching, but recognizing when the problem is upstream.

### Scope issue (rare)
Product QA reveals the intent was ambiguous and the planner made wrong assumptions.

```
Journey "Import" failed:
  User expected CSV import, planner assumed JSON
  → No way to auto-fix without clarification
```

**Action:** Report to user. Don't guess.

### Circuit breaker

- **Fix rounds:** Keep going as long as each round makes measurable progress (fewer failing journeys). Stop if a round fixes nothing new.
- **Replan:** 1 attempt. If the replanned version also fails product QA, the problem is likely scope/ambiguity — report to user.
- **Hard cap:** 3 total product QA invocations (initial + 2 retries) as cost safety net.
- **Scope freeze:** Fix and replan must NOT invent new features. The product spec's non-goals section is the boundary. If a fix requires new scope, report to user instead.

## Outer Loop ↔ Inner Loop Interaction

### Dependencies: two layers, one resolution

The outer loop sets **product-level dependencies** (`depends_on`):
```
Task 1: bookmark CRUD (no deps)
Task 2: search (depends: 1)        ← needs data model from task 1
Task 3: tag system (depends: 1)    ← needs data model from task 1
Task 4: Chrome extension (depends: 1)
```

The inner loop planner adds **file-level constraints** on top:
```
Task 2 & 3: both modify models.py → ADDITIVE → serialize
Task 3 & 4: different files → INDEPENDENT → can parallel
```

**The inner loop's batching is authoritative.** It respects the outer loop's `depends_on` (topological sort) AND adds file-conflict serialization. The outer loop doesn't need to know about batching — it just sets dependencies and calls `run_per()`.

Result:
```
Batch 1: [task 1]           ← foundation
Batch 2: [task 3, task 4]   ← task 3 & 4 both depend on 1, can start together
                               (inner loop may serialize 3 if file conflict with 2)
Batch 3: [task 2]           ← after task 3 (file conflict on models.py)
```

### Fix tasks enter the inner loop normally

When product QA fails and the outer loop creates a fix task:

```python
# Outer loop:
add_task(tasks_file, prompt="Fix: extension calls /bookmarks but API is at /api/bookmarks")
await run_per(config, tasks_file, project_dir)  # inner loop handles it
```

The fix task runs against CURRENT main (all prior tasks already merged). The inner loop sees one pending task, runs it through coding → test → QA → merge. No special handling needed.

### Replan re-executes affected tasks

When the outer loop detects a planning failure:

```python
# Outer loop:
# 1. Mark affected tasks as pending (re-run with new approach)
update_task(tasks_file, "search-task", status="pending")
# 2. Update architecture.md with the revision
(project_dir / "architecture.md").write_text(revised_arch)
# 3. Add new tasks if needed
add_task(tasks_file, prompt="Set up Postgres and migrate from SQLite")
# 4. Re-run inner loop
await run_per(config, tasks_file, project_dir)
```

The inner loop sees: one pending task (re-run) + one new task. It batches by dependencies and executes. Already-passed tasks that aren't affected stay as-is on main.

**Key invariant:** the outer loop interacts with the inner loop through three surfaces:
1. `tasks.yaml` — what to execute
2. `run_per()` — trigger execution
3. Project-root files — shared context (architecture.md, context.md)

It never reaches into the inner loop's internals (no direct calls to `coding_loop`, no manipulation of batches, no custom retry logic). The inner loop's coding agents discover project-root files naturally by exploring the codebase. The outer loop creates them (product planner writes product-spec.md + architecture.md) and maintains them (context updater writes context.md after each task).

## Implementation

### New files
| File | What |
|---|---|
| `otto/product_planner.py` | Classify + plan (one agent call) |
| `otto/product_context.py` | Post-task context updater (one Sonnet call per task) |
| `otto/product_qa.py` | Journey-based verification |
| `otto/outer_loop.py` | Orchestrates: plan → execute → verify → fix/replan |

### Modified files
| File | What |
|---|---|
| `otto/cli.py` | `otto build` command |
| `otto/orchestrator.py` | Call product context updater after each task merges |

### Not modified
| File | Why |
|---|---|
| `otto/planner.py` | Task-level batching unchanged |
| `otto/qa.py` | Task-level QA unchanged |
| `otto/spec.py` | Task-level spec gen unchanged |

### Implementation order

1. **`product_planner.py`** — classify + plan. One agent call, produces spec + tasks. Test with 5-10 intents of varying complexity.
2. **`otto build` in cli.py** — calls planner, shows plan, waits for approval, calls `run_per()`.
3. **`product_context.py`** — post-task context updater. One LLM call after each task merges, updates context.md at project root. Wire into orchestrator.py after merge. Agent discovers it naturally — no prompt injection.
5. **`product_qa.py`** — journey verification agent. Gets product-spec + codebase, tests journeys with browser/API, produces structured verdict.
6. **`outer_loop.py`** — fix/replan loop. Reads product QA verdict, creates fix tasks or replans, re-executes, re-verifies.

Steps 1-3 are a working `otto build` with shared context (no product QA yet). Steps 4-5 close the outer loop.

## What This Does NOT Include

Deliberately excluded. Add only if failure data justifies them.

| Concept | Why excluded |
|---|---|
| **Gates** | Redundant with v4.5 batching + post-batch integration tests |
| **Gate pilot** | Inner loop handles failures well enough. Pilot belongs in outer loop if needed (failure triage in product QA) |
| **Multi-round autonomous fixing** | Max 1 fix round + 1 replan. Not endless patching. |
| **Mandatory architecture.md** | Only when meaningful. Simple projects don't need it. |
| **Agent-to-agent messaging** | Files on disk. No message bus, no discovery board. |
| **Daemon orchestrator** | Deterministic Python + targeted LLM calls at decision points. |
| **`otto chat`** | Can be added later as an interactive mode. Not needed for v1. |

## Open Questions

1. **Planner validation.** Codex suggested a cheap validation pass over the plan (are tasks vertically sliced? are dependencies sane? is each task testable?). Is this worth building, or does user review cover it?

2. **Product-type awareness.** Web apps, CLIs, APIs, libraries, and extensions need different QA strategies. Should the planner tag the product type, and product QA adapt its strategy?

3. **Scope freeze.** How do we prevent the fix/replan loop from inventing new scope? The product spec's non-goals section helps, but is it enough?

4. **Benchmarks.** We need a corpus of build intents with expected outcomes. The weatherapp is one data point. We need 10-20 across different product types and complexity levels.

5. **Cost estimation.** The plan shows "~$5, ~20 min." How accurate can we make this? Historical data from v4.5 runs could inform per-task estimates.
