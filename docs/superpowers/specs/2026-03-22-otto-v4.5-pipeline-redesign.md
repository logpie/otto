# Otto v4.5: Pipeline Redesign — Bare CC First, Otto Layers On Top

**Date**: 2026-03-22
**Status**: Implemented — 2026-03-23
**Prerequisite**: v4 PER orchestrator (parallel batching, serial merge, signal handling)
**Builds on**: v4 design (`2026-03-21-otto-v4-design.md`), workflow audit (`2026-03-22-otto-workflow-audit.md`, `2026-03-22-otto-audit-new-findings.md`)

---

## Core Principle

**Attempt 1 matches bare CC. Otto adds evidence-based verification on top.**

The current pipeline REPLACES bare CC's approach with micromanaged, information-filtered, telephone-game prompts. v4.5 EXTENDS bare CC — same agent, same autonomy, same context — and layers verification + QA on top.

```
Bare CC:    agent works → done
Otto v4.5:  agent works → VERIFY → QA → retry if needed → merge
            ↑ same agent           ↑ otto adds these
              same autonomy
              same context
              no micromanagement
```

**The guarantee**: Attempt 1 = bare CC quality (same agent, same context, no bottleneck). Otto adds a higher ceiling through clean-worktree verification, adversarial QA, and intelligent retry. A verified-and-QA'd candidate is preserved even if later QA disagrees — good work is never discarded.

**Caveat (from Codex review)**: Wrong specs or flaky QA can gate a good bare-CC result. Mitigation: preserve the first verified candidate as a fallback. If QA rejects but verification passed, the candidate is kept for user review rather than silently discarded.

---

## Design Principles (from audit findings)

### 1. Give problems, not instructions
Tell agents WHAT to achieve (goals, context, constraints). Don't tell them HOW (forced workflows, "Do NOT" prohibitions, prescribed approaches). Trust them.

### 2. Pass environmental feedback, not inferred solutions
When information flows between agents or between attempts, pass the RAW DATA (error messages, test output, stack traces). Never pass an upstream agent's interpretation or advice. Let the downstream agent reason from evidence.

### 3. Don't bottleneck information
No truncation of errors, prompts, or context for agents. No system-selected source files. No heuristic filtering of agent output. Give agents full information and let them decide what's relevant.

### 4. Trust agents to self-regulate
No artificial turn caps — agents naturally terminate when done. No forced TDD. No "stay in your lane." No prescribed exploration order. The only hard limit is a 1-hour circuit breaker for pathological cases. Use `max_retries` (default 3) as the primary failure mechanism.

## Pipeline Architecture

### Two parallel agents, then verify/QA

```
ROUND 1 (parallel — bare CC for coding):

  coding_agent ← raw user prompt
                  (bare CC: full autonomy, full tools,
                   has researcher/explorer subagents,
                   no custom system prompt, no micromanagement)

  spec_agent   ← same raw user prompt
                  (full autonomy, explores codebase,
                   researches external APIs/libraries,
                   produces acceptance criteria)

  Both start at t=0. Neither bottlenecks the other.

CONVERGENCE:

  coding agent finishes → candidate commit
  spec agent finishes   → acceptance criteria
  Both outputs converge at verify/QA.

VERIFY + QA:

  verify: run test suite in clean worktree (deterministic)
  QA: adversarial testing against specs (LLM agent)

IF PASS → merge
IF FAIL → retry with:
  - original prompt (same as bare CC)
  - specs (now available — ADDITIVE context, not bottleneck)
  - raw verification/QA errors (environmental feedback)
  - session resume (agent already knows the codebase)
```

### Why two parallel agents?

**Spec gen is NOT an information bottleneck in this design.** The coding agent gets the raw prompt (bare CC), not the spec agent's output. Specs are used for VERIFICATION, not for coding direction.

The spec agent and coding agent independently interpret the user's intent. When they DISAGREE, that's the most valuable signal — it reveals ambiguity that neither alone would catch. Verification surfaces these disagreements, and retry resolves them.

**Spec gen is always worth running because it's free (parallel).** For clear tasks, specs restate the prompt in 1-2 turns (~10s). For vague tasks, specs add essential verification criteria. Either way, no latency cost.

### When is spec gen skipped?

Only when the user already provided specs via `otto add --spec` or the task already has specs in `tasks.yaml`. If specs exist, only the coding agent runs in round 1.

## Design Invariant: Original Prompt Always Available

The user's original task prompt (`task["prompt"]`) is the source of truth for intent. It must be available to EVERY component:

| Component | Gets original prompt? | How |
|---|---|---|
| Coding agent (round 1) | **Yes** — it IS the prompt | `coding_prompt = prompt` |
| Coding agent (retry) | **Yes** — base of retry prompt | `retry_prompt = prompt + specs + errors` |
| Spec agent | **Yes** — generates specs from it | `generate_spec(prompt, project_dir)` |
| QA agent | **Yes** — checks prompt intent | Explicitly passed: `original_prompt=prompt` |
| Planner/replanner | **Yes** — full prompts, not truncated | `task["prompt"]` (no [:100] truncation) |
| Cross-task learnings | **No** — learnings are observations, not prompts | Factual only, with provenance |

## Per-Task Execution

```python
async def run_task(task, config, project_dir, context, sem, telemetry):
    """Execute a single task: bare CC + verify + QA + retry."""
    worktree = await asyncio.to_thread(setup_worktree, project_dir, task["key"])
    prompt = task["prompt"]  # ORIGINAL PROMPT — never modified, always available
    spec = task.get("spec")
    session_id = None
    last_error = None
    last_error_source = None
    total_cost = 0.0
    task_start = time.monotonic()

    # Fire spec gen in background if no specs yet.
    # Gated by semaphore, with timeout and error handling.
    spec_task = None
    if not spec:
        async def _spec_with_timeout():
            try:
                async with sem:
                    return await asyncio.wait_for(
                        generate_spec(prompt, project_dir),
                        timeout=300,  # 5 min max for spec gen
                    )
            except (asyncio.TimeoutError, Exception):
                return None  # spec gen failed — proceed without specs
        spec_task = asyncio.create_task(_spec_with_timeout())

    for attempt in range(config.get("max_retries", 3) + 1):
        # Ensure specs are available before attempt 2+
        # (Codex: retry without specs is a timing race)
        if attempt > 0 and spec_task and not spec:
            spec = await spec_task
            spec_task = None

        # Build the prompt for this attempt
        if attempt == 0:
            # ROUND 1: Bare CC — raw prompt + cross-task learnings
            coding_prompt = prompt
            if context.learnings:
                coding_prompt += f"\n\nFactual observations from prior tasks:\n"
                coding_prompt += "\n".join(
                    f"- [{l['source']}] {l['text']}" for l in context.learnings
                )
        else:
            # ROUND 2+: Additive context — specs + raw errors + learnings
            coding_prompt = prompt
            if spec:
                coding_prompt += f"\n\nAcceptance criteria (satisfy [must], exceed where helpful):\n"
                coding_prompt += format_spec(spec)
            if last_error:
                coding_prompt += f"\n\nPrevious attempt failed."
                coding_prompt += f"\n  Source: {last_error_source}"
                coding_prompt += f"\n  Raw output:\n{last_error}"
            if context.learnings:
                coding_prompt += f"\n\nFactual observations from prior tasks:\n"
                coding_prompt += "\n".join(
                    f"- [{l['source']}] {l['text']}" for l in context.learnings
                )

        coding_prompt += "\n\nDo not create git commits."

        # Run coding agent
        async with sem:
            coding_result = await run_coding_agent(
                coding_prompt, worktree, config, session_id=session_id
            )

        total_cost += coding_result.cost
        session_id = coding_result.session_id

        # BUILD CANDIDATE
        candidate = await asyncio.to_thread(build_candidate_commit, worktree, task)
        if not candidate:
            last_error = "No code changes produced."
            last_error_source = "coding"
            continue

        # VERIFY — clean worktree, deterministic
        verify = await asyncio.to_thread(
            run_verification, worktree, candidate, config
        )
        telemetry.log(VerifyResult(task["key"], verify.passed))

        if not verify.passed:
            last_error = verify.failure_output  # RAW output, full, no truncation
            last_error_source = "verify"
            await asyncio.to_thread(reset_to_base, worktree)
            continue

        # ANCHOR verified candidate as durable git ref
        # (Codex: SHAs without refs can become dangling after reset)
        ref_name = f"refs/otto/candidates/{task['key']}/attempt-{attempt}"
        await asyncio.to_thread(
            lambda: subprocess.run(
                ["git", "update-ref", ref_name, candidate],
                cwd=worktree, capture_output=True,
            )
        )

        # Await specs before QA (if still generating)
        if spec_task and not spec:
            spec = await spec_task
            spec_task = None

        # QA — risk-based tiering
        if spec:
            spec_test_map = extract_spec_test_mapping(worktree, spec)
            diff_info = await asyncio.to_thread(get_diff_info, worktree)
            qa_tier = determine_qa_tier(task, spec, attempt, diff_info, spec_test_map)

            if qa_tier >= 1:
                async with sem:
                    qa = await run_qa_agent(
                        task, spec, worktree, config,
                        original_prompt=prompt,  # QA checks prompt intent
                        diff=diff_info.full_diff,  # actual diff, not --stat
                        tier=qa_tier,
                        focus_items=[i for i in spec if not spec_test_map.get(i["text"])],
                    )
                total_cost += qa.cost
                if not qa.must_passed:
                    last_error = qa.raw_report
                    last_error_source = "qa"
                    await asyncio.to_thread(reset_to_base, worktree)
                    continue

        # SUCCESS
        return TaskResult(
            task_key=task["key"], success=True,
            commit_sha=candidate, worktree=str(worktree),
            cost_usd=total_cost,
            duration_s=time.monotonic() - task_start,
        )

    # All retries exhausted.
    # Find best verified candidate (anchored as git ref).
    # NEVER auto-merge a candidate with must failures.
    # Surface as a review branch for the user.
    best_ref = find_best_candidate_ref(worktree, task["key"])

    return TaskResult(
        task_key=task["key"], success=False,
        error=last_error, cost_usd=total_cost,
        duration_s=time.monotonic() - task_start,
        review_ref=best_ref,  # user can: git show refs/otto/candidates/<key>/attempt-N
    )
```

## Agent Configurations

### Coding Agent

```python
agent_opts = ClaudeAgentOptions(
    permission_mode="bypassPermissions",
    cwd=str(worktree),
    setting_sources=["user", "project"],
    env=os.environ,
    # NO custom system_prompt — use CC's default

    # NO forced TDD, no "Do NOT", no micromanagement
    agents={
        "researcher": AgentDefinition(
            description="Research APIs, read docs, investigate approaches",
            prompt="Investigate the topic and report findings.",
        ),
        "explorer": AgentDefinition(
            description="Search codebase for patterns, find relevant files",
            prompt="Search for relevant code and report what you find.",
        ),
    },
)
```

**What's different from v3/v4:**
- No `CODING_SYSTEM_PROMPT` (was 30 lines of micromanagement)
- No `max_turns` (was 200 — agent finishes naturally, 1hr circuit breaker only)
- No pre-loaded source files (agent explores what IT thinks is relevant)
- No "Do NOT re-read", "Do NOT start dev servers", "Do NOT create git commits" — only the last one is kept (necessary constraint: `prompt += "Do not create git commits."`)
- No forced TDD ("WRITE TESTS FIRST")
- No "stay in your lane"

The coding agent is bare CC with one constraint: don't commit (otto handles git).

On retry (when specs are available), the prompt says: "Satisfy all [must] items. [should] items are suggestions — use your judgment." — specs are a floor, not a ceiling.

### Spec Agent

```python
agent_opts = ClaudeAgentOptions(
    permission_mode="bypassPermissions",
    cwd=str(project_dir),
    setting_sources=["user", "project"],
    env=os.environ,
    system_prompt=SPEC_SYSTEM_PROMPT,
    # NO max_turns — agent finishes naturally
)
```

**System prompt — two binding levels (RFC 2119: MUST/SHOULD):**

```
You produce acceptance criteria for a coding task. Your output sets
the minimum bar — the coding agent may EXCEED it but must not
violate constraints.

Two binding levels:
  [must]   — gating. Task fails QA if not met. Agent may exceed.
             Use for: functional requirements, measurable constraints,
             error handling, API contracts, security, performance limits.
  [should] — non-gating. QA notes but does not block merge.
             Use for: UX preferences, style, quality suggestions,
             "nice to have" behaviors.

Rules:
- Describe OBSERVABLE BEHAVIOR — what the user sees and experiences.
- Do not prescribe implementation (no file names, no code patterns).
- Preserve every user constraint exactly. Do not weaken thresholds.
- Produce as many or as few criteria as the task warrants.
- Explore the codebase and research external APIs/libraries/services
  as needed to write accurate, grounded criteria.
- When the prompt is ambiguous, prefer [should] over [must].
  Don't promote ambiguous preferences to hard constraints.
  Don't invent product decisions.

Output format — one item per line:
  [must] rate-limited requests return HTTP 429
  [must] response completes within 200ms p95
  [must] UI shows wind chill formula with current values
  [should] prefer inline explanation of each contributing factor
  [should] credit source model if attribution is available
```

**Key design decision: specs are a FLOOR, not a CEILING.**

The v3/v4 spec format (`[verifiable]`/`[visual]` with "meet ALL exactly, nothing more") turned specs into a ceiling — the coding agent wouldn't go beyond the checklist. The wind chill example showed bare CC producing better UX because it used judgment instead of following a spec literally.

The two-level model (RFC 2119) fixes this:
- `[must]` items are the minimum bar — gating, but agent may exceed
- `[should]` items are suggestions — non-gating, agent uses judgment
- The coding agent is told "satisfy [must], use judgment beyond that"
- QA gates on [must] only. [should] items produce quality notes, never block merge.

**What's different from current:**
- Replaced `[verifiable]`/`[visual]` with `[must]`/`[should]` (two levels, not three)
- Specs are a floor, not a contract — agent may exceed [must] items
- Removed "5-8 acceptance criteria" cap → "as many or as few as needed"
- Removed `test_hint` from output format
- Removed prescribed workflow → agent decides how to explore
- Added explicit research encouragement
- Added ambiguity discipline: "prefer [should] over [must] when ambiguous"

### QA Agent

```python
agent_opts = ClaudeAgentOptions(
    permission_mode="bypassPermissions",
    cwd=str(worktree),
    setting_sources=["user", "project"],
    env=os.environ,
    system_prompt=QA_SYSTEM_PROMPT,
    # NO max_turns — bounded by qa_timeout (or 1hr circuit breaker)
)
```

**System prompt — two-level verdict + browser testing:**

```
You are an adversarial QA tester. Test the implementation against
the acceptance criteria and the original task prompt.

Binding levels:
- [must] items: a failure here blocks merge. Test the hardest cases.
- [should] items: note whether followed, but do not block merge.

Also check:
- Does the implementation contradict the ORIGINAL task prompt?
- Does it break existing functionality?
- Did the agent add improvements beyond the spec? Note them positively.

Choose your testing approach based on what the task needs:
- API/CLI tasks: curl, subprocess, script-based checks
- SSR web apps: curl for content, browser for interactivity/layout
- SPA/client-side apps: browser is essential (curl sees empty div)
- Visual/UX tasks: browser + screenshot is the only way to verify

Write your verdict to the output file as JSON:
{
  "must_passed": true/false,
  "must_items": [
    {"criterion": "...", "status": "pass/fail", "evidence": "..."}
  ],
  "should_notes": [
    {"criterion": "...", "observation": "...", "screenshot": "path or null"}
  ],
  "regressions": [],
  "prompt_intent": "Implementation matches/diverges from original prompt because...",
  "extras": ["Agent added contributing factor explanations — improves UX"]
}
```

**Key design: QA gates on [must], reviews [should], checks original prompt.**

- `must_passed` is the gating decision. Only [must] failures block merge.
- `should_notes` are non-gating quality observations.
- `prompt_intent` verifies against the original prompt (not just the spec) — catches cases where the spec is wrong or incomplete but the coding agent correctly interpreted the user's intent.
- `extras` acknowledges improvements the agent made beyond the spec.

**Browser testing in v4.5:**

QA gets chrome-devtools MCP when specs contain visual/UI-related items. Detection: keyword match on spec item text ("UI", "layout", "style", "visual", "responsive", "screenshot", "animation").

```python
# QA agent configuration
qa_mcp_servers = {}
if has_visual_or_ui_specs(spec):
    chrome_config = load_chrome_devtools_config()
    if chrome_config:
        # Headless, 1280x720, shared profile (avoid 1.9G clone copies)
        chrome_config["args"] += ["--headless", "--viewport", "1280x720"]
        qa_mcp_servers["chrome-devtools"] = chrome_config

qa_opts = ClaudeAgentOptions(
    permission_mode="bypassPermissions",
    cwd=str(worktree),
    setting_sources=["user", "project"],
    env=os.environ,
    system_prompt=QA_SYSTEM_PROMPT,
    mcp_servers=qa_mcp_servers,
)
```

The QA agent decides whether to use browser testing. If it needs to `npm run dev` and take screenshots, it does that. No "Do NOT start servers" prohibition.

Screenshots are saved to `otto_logs/{task_key}/` and referenced in the QA verdict JSON. Users see them via `otto show <task>`.

**What's different from current:**
- Removed "Do NOT re-run existing tests" → agent decides
- Removed "Do NOT build the project or start dev servers" → agent decides
- Removed "Be FAST. Target 2-3 minutes" → bounded by timeout, not prompt instruction
- Removed prescribed testing layers (static → functional → browser) → agent decides approach
- Removed "Do NOT rebuild the project" → agent decides

The QA agent gets the acceptance criteria and the diff. It decides how to test.

## Retry Design: Environmental Feedback Only

On failure, the retry prompt contains:
1. **Original prompt** (same as round 1)
2. **Specs with binding levels** (now available from parallel spec gen — `[must]`/`[should]`)
3. **Raw error output with provenance** (source: verify/qa, full output, NO truncation)
4. **Cross-task learnings** (factual observations with provenance, e.g., "[task-1] project uses ESM modules")
5. **Framing**: "Satisfy all [must] items. Use judgment to exceed where helpful."

What it does NOT contain:
- Hints ("try moving func inside lock") — telephone game
- Prescribed diagnosis ("Is it a code bug or a test bug?") — micromanagement
- Approach suggestions ("use a different algorithm") — system substituting judgment
- "Nothing more" / "exactly" language — specs are a floor, not a ceiling

```python
# RETRY PROMPT (v4.5)
retry_prompt = task["prompt"]

if spec:
    retry_prompt += f"\n\nAcceptance criteria:\n{format_spec(spec)}"

if last_error:
    # RAW error — full output, no truncation, no interpretation
    retry_prompt += f"\n\nPrevious attempt failed. Raw output:\n{last_error}"

if context.learnings:
    # Factual observations, not advice
    retry_prompt += f"\n\nFactual observations from prior tasks:\n"
    retry_prompt += "\n".join(f"- {l}" for l in context.learnings)
```

## Spec Is Not a Bottleneck (Architectural Guarantee)

In v4.5, specs CANNOT bottleneck the coding agent because of how information flows:

```
v3/v4:   spec → coding agent (spec IS the input — BOTTLENECK)
v4.5:    prompt → coding agent (raw, unfiltered — bare CC)
         spec  → QA agent (independent verification — NOT A BOTTLENECK)
```

| Phase | Does coding agent see specs? | Can specs constrain creativity? |
|---|---|---|
| Round 1 | **No** — gets raw prompt only | **No** — agent uses bare CC judgment |
| QA | N/A — QA uses specs, not coding agent | Specs are a floor: [must] gating, [should] = non-gating |
| Round 2+ retry | **Yes** — specs added as context | **Minimal** — framed as "satisfy + exceed where helpful" |

Specs moved from INPUT (directing the coding agent) to VERIFICATION (evaluating the output). The coding agent's creative judgment is never constrained by specs.

**Remaining risk and mitigation:**

| Risk | Mitigation |
|---|---|
| Wrong specs → QA rejects good code | Verified candidate preserved as fallback for user review |
| Over-specific specs | Only [must] is gating. [should] = non-gating |
| Missing specs | QA checks against ORIGINAL PROMPT too, not just spec items |

## QA Tiering: Risk-Based, Not Spec-Count-Based

Don't classify tasks as "simple vs complex." Classify **residual risk after verification passes.** (From Codex review: `spec_count <= 3` is artificial — a 2-spec task can be complex, a 10-spec task can be simple.)

### Three QA tiers

```
Tier 0 — SKIP QA: All of these must be true:
  - First attempt, verify passed cleanly
  - Diff is local/patterned (not cross-cutting)
  - Every [must] item has a corresponding test (spec-to-test mapping)
  - No auth, crypto, persistence, external API, or visual/UI changes
  → Cost: 0s

Tier 1 — TARGETED QA: Any of these:
  - Some [must] items lack test coverage (spec-to-test gap)
  - Diff is cross-cutting (touches many files/modules)
  - Task introduced new behavior with thin tests
  → QA focuses on unmapped [must] items + adversarial probes
  → Cost: 30-60s (curl/script-based, no browser)

Tier 2 — FULL QA: Any of these:
  - Touches auth, permissions, crypto, migrations, external APIs
  - Has [must] performance items
  - Has visual/UI [should] items that need browser verification
  - SPA/client-side app (curl can't verify JS-rendered content)
  - Previous attempt failed (retry)
  → Full adversarial QA agent with browser if needed
  → Cost: 60-160s
```

The tier is determined AFTER verification passes, based on what was actually built — not upfront by the planner.

### Test ownership model (from Codex review)

The coding agent and QA agent serve different roles — they're not duplicative:

| | Coding agent | QA agent |
|---|---|---|
| **Question** | "Does my implementation work?" | "Does it meet the spec + prompt intent?" |
| **Perspective** | Inside-out (code-aware) | Outside-in (black-box) |
| **Artifacts** | Repo tests (unit, integration) | Structured JSON verdict + targeted probes |
| **Catches** | Regressions, anticipated edge cases | Spec mismatches, MISSED edge cases, prompt contradictions |

**Spec-to-test traceability**: The coding agent should output a mapping — which test covers which [must] item. If every [must] item has a test, QA can be lighter (Tier 0 or 1). If items lack tests, QA focuses there.

```python
# Coding agent's test mapping (written to task notes)
spec_to_test:
  "[must] returns 429 for rate-limited requests": "test_rate_limit.py::test_over_limit"
  "[must] response < 200ms": "test_perf.py::test_response_time"
  "[must] shows wind chill formula": "test_weather.py::test_formula_displayed"
  "[must] handles negative wind speed": null  # NO TEST — QA should focus here
```

**When QA finds a bug**: The repro command gets promoted into a persistent regression test before merge. Over time, repeated QA catches graduate into deterministic Tier 1 checks — QA gets cheaper as the test suite grows.

```python
def determine_qa_tier(task, spec, attempt, diff_stat, spec_test_mapping):
    """Determine QA tier based on residual risk after verification."""

    diff_files = diff_stat.get("files", [])

    # Tier 2: high-risk domains
    HIGH_RISK_PATTERNS = ["auth", "crypto", "permission", "migration",
                          "payment", "security", "token", "session"]
    if any(pattern in f.lower() for f in diff_files for pattern in HIGH_RISK_PATTERNS):
        return 2

    # Tier 2: visual/UI specs (need browser), SPA apps, or retries
    has_visual = any(item.get("binding") == "should"
                     and any(kw in item["text"].lower()
                             for kw in ("ui", "layout", "style", "visual", "responsive"))
                     for item in spec)
    is_spa = any(f.endswith((".jsx", ".tsx", ".vue", ".svelte")) for f in diff_files)
    if has_visual or is_spa or attempt > 0:
        return 2

    # Tier 1: unmapped [must] items or cross-cutting changes
    unmapped = [item for item in spec
                if item.get("binding") == "must"
                and not spec_test_mapping.get(item["text"])]
    if unmapped or len(diff_files) > 5:
        return 1

    # Tier 0: every [must] item has a test, local change, first attempt
    return 0
```

## Multi-Task Design (from Codex review)

### Integration with v4 PER Orchestrator

v4.5 changes per-task execution. The v4 PER orchestrator wraps it:

```
PLAN (deterministic topological sort via default_plan())
  → EXECUTE batch (parallel tasks, each using v4.5 per-task loop)
    → POST-BATCH VERIFY (re-run tests after all merges to catch semantic conflicts)
      → REPLAN at batch boundary (with raw TaskResults)
        → failed tasks get a retry lane in the next batch
```

### Cross-task learnings on first attempt

Codex identified a regression: if round 1 coding agent gets ONLY the raw prompt, later tasks lose learnings from earlier tasks.

**Fix:** Task 1 gets bare prompt. Later tasks get bare prompt + factual cross-task learnings WITH PROVENANCE. Each learning has a source task and type:

```python
# Learnings are structured, not flat strings
@dataclass
class Learning:
    text: str       # "project uses ESM modules"
    source: str     # "task-1" (which task discovered this)
    kind: str       # "observed" (factual) vs "inferred" (interpretation)

# Only "observed" learnings are passed to coding agents.
# "inferred" learnings are logged for debugging but not injected.
```

This prevents the telephone game risk Codex identified: "project uses ESM" is observed (factual). "Middleware pattern is X" is inferred (interpretation that could steer the agent wrong). Only factual observations are passed forward.

### Correlated batch failure recovery

If all tasks in a batch fail with correlated symptoms (e.g., wrong framework version), per-task retries are wasted — each task independently exhausts 3 retries before the replanner sees the pattern.

**Fix:** If ≥2 tasks in a batch fail with similar errors (heuristic: shared substrings in error output), stop per-task retries early and escalate to batch replan. The replanner sees the correlated failure and can inject a cross-cutting learning before retrying.

```python
# In orchestrator, after batch execution:
failures = [r for r in results if not r.success]
if len(failures) >= 2 and errors_are_correlated(failures):
    # Don't retry individually — escalate to replan with correlated failure context
    context.add_learning(f"Correlated failure in batch: {common_error_pattern(failures)}")
    # Replan will produce a new batch with the learning injected
```

### Safe batch integration (from Codex review)

Codex identified: merging directly to the default branch then verifying is unsafe — if tests fail, main is already broken.

**Fix:** Merge to a temporary integration branch. Verify there. Fast-forward main only if tests pass.

```python
# Create scratch integration branch from current default
integration_branch = f"otto/integration-{batch_id}"
subprocess.run(["git", "checkout", "-b", integration_branch, default_branch], ...)

# Serial merge of batch tasks (rebase-before-merge)
for result in batch_results:
    if result.success:
        rebase_and_merge(project_dir, result.worktree, result.task_key, integration_branch)

# Verify the INTEGRATED codebase (catches semantic conflicts)
integration_verify = await asyncio.to_thread(
    run_verification, project_dir, "HEAD", config
)

if integration_verify.passed:
    # Safe to fast-forward main
    subprocess.run(["git", "checkout", default_branch], ...)
    subprocess.run(["git", "merge", "--ff-only", integration_branch], ...)
    subprocess.run(["git", "branch", "-d", integration_branch], ...)
else:
    # Integration failure — don't touch main
    telemetry.log(IntegrationFailure(batch_keys, integration_verify.failure_output))
    # Feed to replanner: which tasks conflict?
    subprocess.run(["git", "checkout", default_branch], ...)
    subprocess.run(["git", "branch", "-D", integration_branch], ...)
    # Replanner gets the failure and can retry tasks individually
```

### Auto-dependency detection (replaces manual `depends_on`)

Users never need to specify `depends_on` manually. The existing architect agent already does this via `file-plan.md`, but the full architect (7 output files, 15 turns) is overkill — most of its output is unused (audit C1).

**v4.5: scope the architect down to file-plan only.** One focused query() call, ~5 turns:

```python
async def predict_file_overlaps(tasks, project_dir):
    """Predict which tasks touch the same files. Returns dependency pairs.

    Runs only for multi-task (≥2 tasks). Single-task skips this.
    """
    if len(tasks) < 2:
        return []

    prompt = f"""Given these tasks, predict which files each will modify.
    Explore the project structure as needed.

{format_tasks(tasks)}

Output YAML:
tasks:
  - id: 1
    predicted_files: [path/to/file.py]
  ...
recommended_dependencies:
  - from: 2
    depends_on: 1
    reason: "both modify path/to/file.py"
"""
    result = await query(prompt=prompt, options=ClaudeAgentOptions(
        cwd=str(project_dir),
        permission_mode="bypassPermissions",
    ))
    return parse_file_plan_from_output(result)
```

This runs at the start of `otto run`, before batching. Dependencies feed into `default_plan()` topological sort. The user just does `otto add "task 1"` + `otto add "task 2"` + `otto run` — dependencies are handled automatically.

### Planner simplification

With auto-deps handling batching and hints/effort/skip_qa removed, the LLM planner's remaining value is minimal. `default_plan()` (deterministic topological sort with auto-detected dependencies) handles everything.

**Decision:** Use `default_plan()` by default. The LLM planner is available via `otto run --planner` for complex multi-task runs where semantic ordering matters. Evaluate whether anyone uses it after v4.5 ships.

## Open Questions

1. **Coding agent minimal system prompt?** Current design says "no custom system prompt." CC's default prompt includes general coding guidelines. Consider whether a TINY role prompt adds value or if CC's default is sufficient. The only content that MUST be in the prompt: "do not create git commits."

2. **Spec agent model:** Both spec gen and coding consume subscription quota in parallel. Should spec gen use a cheaper model? `ClaudeAgentOptions.model` supports per-agent model selection.

3. **`otto plan` command:** Show execution plan + spec preview without executing. Deferred or v4.5?

4. **Correlated failure early-stop:** Currently per-task retries exhaust before batch-level correlation detection. Future: limit per-task retries to 1 on multi-task, let replanner decide retry strategy. Deferred — requires refactoring per-task retry ownership.

5. **QA infra failure policy:** What happens if chrome-devtools crashes mid-QA, or the dev server won't start? Currently undefined. Likely: treat as QA inconclusive, merge if verify passed, flag for user review.

## Deferred to v4.6: Spec Timing (Parallel vs Serial)

v4.5 uses parallel spec+coding (specs in background, used for QA/retry). This was debated extensively:

**The tension:**
- Parallel saves 2-5 min of latency (spec gen can take 4+ min, not 30-40s as originally estimated)
- Serial gives coding agent specs on round 1 → higher first-attempt pass rate for vague tasks
- The [must]/[should] floor model means specs are additive, not constraining → serial is safe
- But there's a behavioral subtlety: LLMs focus on structured [must] lists, reducing creative judgment even when told "exceed where helpful"
- Codex recommended serial as default ("hiding specs from round 1 is information deprivation once specs are a floor")

**Why deferred:** The right answer depends on empirical data we don't have yet:
- How long does spec gen ACTUALLY take across different project sizes?
- How often does round 1 fail due to missing specs vs pass without them?
- Does the [must]/[should] framing actually prevent the ceiling effect in practice?
- Does the latency cost of serial (2-5 min idle) outweigh the retry savings?

**v4.6 plan:** Ship v4.5 with parallel. Collect data on:
1. Spec gen latency (p50, p95 across tasks)
2. First-attempt pass rate (with and without specs)
3. Retry rate attributable to missing specs
4. Whether [must]/[should] framing prevents ceiling effect (compare creative quality vs bare CC)

If data shows serial is better → v4.6 makes serial the default with `--parallel-spec` opt-in.
If data shows parallel is better → keep parallel, close the question.

Either way: `otto run --specs-first` flag for users who want serial. `spec_timing: serial|parallel` in otto.yaml for project-level config.

1. **Make `otto add` instant** — remove spec gen from add, store raw prompt only.
