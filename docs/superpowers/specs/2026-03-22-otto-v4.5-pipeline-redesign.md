# Otto v4.5: Pipeline Redesign — Bare CC First, Otto Layers On Top

**Date**: 2026-03-22 (design) → 2026-03-24 (v4.5 tagged and released)
**Status**: Released as v4.5. Per-task pipeline fully implemented. Verified with 422 tests and 8+ weatherapp e2e tasks.
**Previous version**: `2026-03-22-otto-v4.5-pipeline-redesign.v1-draft.md`

---

## Core Principle

**Attempt 1 matches bare CC. Otto adds evidence-based verification on top.**

```
Bare CC:    agent works → done
Otto v4.5:  agent works → CLEAN TEST → SPEC VERIFICATION → retry if needed → merge
            ↑ same agent              ↑ otto adds these
              same autonomy
              same context
              no micromanagement
```

Attempt 1 = bare CC quality (same agent, same context, no bottleneck). Otto adds a higher ceiling through clean-worktree testing, adversarial spec verification, and intelligent retry. Verified candidates are preserved as durable git refs — good work is never discarded.

## Terminology

| Old name | What it actually does | v4.5 name |
|---|---|---|
| "Verify" | Runs test suite in clean worktree | **Clean test** — regression check, not spec verification |
| QA agent | LLM checks [must]/[should] items + prompt intent | **Spec verification** — the real independent quality gate |
| "Pilot" (v3) | Long-running LLM driving every step | **Planner** — called at batch boundaries only |
| "Hints" | LLM-interpreted retry advice | **Raw error feedback** — environmental data, not advice |

**The quality layers:**
- **Clean test** (per-task, cheap, deterministic): "Do tests pass in clean env?" Catches regressions. Does NOT catch spec mismatches.
- **Spec verification** (per-batch, LLM): "Does the implementation meet [must] criteria + original prompt intent?" The real quality gate.
- **Integration test** (per-batch, cheap): "Does the merged codebase still work?" Catches cross-task conflicts.

Note: terminology rename is conceptual — code still uses `verify`/`qa` in variable names.

---

## Design Principles

1. **Give problems, not instructions** — tell agents WHAT to achieve, not HOW. Trust them.
2. **Pass environmental feedback, not inferred solutions** — raw errors with provenance, never hints or interpreted advice.
3. **Don't bottleneck information** — no truncation for agents, no system-selected files, no heuristic filtering.
4. **Trust agents to self-regulate** — no turn caps (1hr circuit breaker only), no forced TDD, `max_retries` as primary failure mechanism.
5. **Specs are a floor, not a ceiling** — [must] items are the minimum bar, agents may exceed. QA also checks original prompt intent.
6. **Stress-test harness assumptions** — every component encodes a model limitation assumption. Periodically test if it's still needed. (From Anthropic harness design article.)

## Spec Format: [must] / [should]

Two binding levels (RFC 2119):

```
[must]   — gating. Task fails spec verification if not met. Agent may exceed.
[should] — non-gating. Noted in verdict but doesn't block merge.
```

Spec agent prompt: "prefer [should] over [must] when ambiguous. Don't invent product decisions."

Non-verifiable (subjective) items get a ◈ marker: `[must ◈]`, `[should ◈]`.

**Specs are a FLOOR, not a CEILING.** The coding agent is told "satisfy [must], exceed where helpful." QA checks original prompt intent too — catches cases where specs missed something the coding agent correctly inferred.

## Per-Task Pipeline (Implemented)

```
ROUND 1 (parallel):
  coding_agent ← raw prompt + cross-task learnings (bare CC)
  spec_agent   ← raw prompt (background, 5-min timeout)

  coding finishes → candidate commit → clean test (tests in clean worktree)
  spec finishes → specs available for QA

  clean test passes → await specs → spec verification (QA)
  spec verification passes → merge

IF CLEAN TEST FAILS:
  retry with: raw prompt + raw error (source: clean_test) + specs (if ready)

IF SPEC VERIFICATION FAILS:
  retry with: raw prompt + specs + raw QA findings (source: qa)

Candidate anchored as refs/otto/candidates/{key}/attempt-{N}
Best candidate surfaced as review_ref if all retries exhausted
```

## Spec Verification (QA) Scope

The spec items are the MINIMUM QA checks. QA should also try to break things the spec didn't anticipate:

| QA responsibility | Gating? | Verdict field | Status |
|---|---|---|---|
| [must] items pass | **Yes** | `must_items` | **Implemented** |
| [should] items observed | No | `should_notes` | **Implemented** |
| Original prompt intent | Advisory | `prompt_intent` | **Implemented** |
| Regressions | **Yes** (critical) | `regressions` | **Implemented** |
| Extras acknowledged | No | `extras` | **Implemented** |
| Adversarial edge cases | No | `adversarial` | **Not implemented** — add to QA prompt |
| Security review | Optional (future) | `security` | **Not implemented** |
| Code quality observations | No | `code_quality` | **Not implemented** — add to QA prompt |

QA verdict JSON:
```json
{
  "must_passed": true/false,
  "must_items": [{"criterion": "...", "status": "pass/fail", "evidence": "..."}],
  "should_notes": [{"criterion": "...", "observation": "...", "screenshot": "path or null"}],
  "regressions": [],
  "prompt_intent": "...",
  "extras": ["..."]
}
```

Browser testing: QA gets chrome-devtools MCP for Tier 2 (visual/SPA/high-risk tasks). Agent decides whether to use curl or browser.

## Risk-Based QA Tiering (Implemented)

| Tier | When | What QA does | Cost |
|---|---|---|---|
| Tier 0 — skip | First attempt, local diff, all [must] mapped to tests | No QA | 0s |
| Tier 1 — targeted | Unmapped [must] items, cross-cutting diff | Focus on gaps | 30-60s |
| Tier 2 — full | Auth/crypto/payments, visual/SPA, retries | Full adversarial + browser | 60-160s |

Tiering is decided AFTER clean test passes, based on what was actually built.

Future: move to per-batch QA (one session covering all tasks) instead of per-task.

## Modular Harness: Toggle-able Components (Not Implemented)

From Anthropic harness design: *"every component encodes an assumption about what the model can't do on its own."*

```yaml
# otto.yaml — planned, not yet implemented
harness:
  spec_gen: true          # false → coding agent works from raw prompt, QA tests against prompt
  qa: true                # false → merge after clean test passes
  verify_worktree: true   # false → trust agent's in-session test runs
  retry: true             # false → one attempt only
  auto_deps: true         # false → no file-plan detection
```

Stress-test: disable one component at a time, run pressure suite, compare results. If disabling doesn't degrade quality, the model has outgrown it.

## QA Prompt Tuning (Not Started)

From Anthropic: *"Out of the box, Claude is a poor QA agent. I watched it identify legitimate issues, then talk itself into deciding they weren't a big deal."*

Process (iterative, not one-time):
1. Ship with minimal QA prompt (~15 lines) — **done**
2. Run pressure test suite, collect `qa-verdict.json` files
3. Review: false positives, false negatives, self-talk-downs
4. Update QA prompt to address patterns
5. Re-run, verify improvement
6. Track prompt version with changelog

Also monitor for **QA self-talk-down**: agent identifies issue → reasons it away → approves. The fail-fast logic (stop after [must] failure) partially mitigates this.

## Deferred to v4.6: Spec Timing (Parallel vs Serial)

v4.5 uses parallel (specs in background). Codex recommended serial ("hiding specs is information deprivation once specs are a floor"). The right answer depends on empirical data:

- Spec gen latency (p50, p95 — is it 30s or 5min?)
- First-attempt pass rate (with and without specs)
- Whether [must]/[should] prevents the ceiling effect

Ship v4.5 with parallel. Collect data. v4.6 decides.

`--specs-first` flag and `spec_timing: serial|parallel` config planned but not implemented.
